## The blocker (recurrence of the R14 PAT-scope rejection)

```
! [remote rejected] main -> main
  (refusing to allow a Personal Access Token to create or update
   workflow `.github/workflows/publish.yml` without `workflow` scope)
```

GitHub refuses, atomically, any push that creates or updates a file
under `.github/workflows/` if the PAT does not carry the `workflow`
scope (classic PAT) or *Actions: Read and write* (fine-grained PAT).
The PAT in `.gh-token` still does not have it — the same root cause
as 2026-05-10 / 2026-05-11 / 2026-05-12 / 2026-05-13. Nothing else
in the run failed.

The R14a design note in `docs/design.md` explicitly anticipated this:

> Still under *Up next* in the roadmap. The split exists because
> pushing a file under `.github/workflows/` requires a GitHub PAT
> carrying the `workflow` scope (classic) or *Workflows: Read and
> write* (fine-grained); the daily-task PAT currently does not
> carry that scope, so R14b cannot land via the scheduled run.

Today's run is the empirical confirmation.

## Decision

- The local R14b commit (`feat(release): R14b — PyPI publish
  workflow file`) was rolled back with `git reset --hard
  origin/main` after the push rejection, so `main` does not
  falsely claim R14b shipped and the ROADMAP / CHANGELOG /
  release-doc edits do not get partially applied.
- This work-log is the **only** file in today's commit. A
  docs-only push does not touch `.github/workflows/` and therefore
  does not trip the scope check — so today's blocker is at least
  visibly recorded on `origin/main`, even when R14b itself cannot
  land.
- The prepared R14b files are recreatable from this work-log; the
  full diff is small (workflow file 100 lines, test file 316 lines,
  plus small doc edits) so re-staging once the PAT is fixed is
  not expensive.

## Action required by Howard (fifth reminder)

The same one-line change documented since 2026-05-10 will unblock
R14b:

- **Classic PAT** (`ghp_…`): tick the **`workflow`** scope (sits
  under `repo`).
- **Fine-grained PAT**: set **Workflows: Read and write** under
  *Repository permissions* in addition to the existing
  *Contents: Read and write*.

Then replace the `GITHUB_TOKEN=...` line in `.gh-token` with the
new value. Nothing else in the project needs to change. The next
scheduled run will pick R14b from "Up next" and ship the same set
of files in one push.

A GitHub MCP connector with `workflow` permission for repo writes
remains the documented alternative if the PAT scope cannot be
updated.
