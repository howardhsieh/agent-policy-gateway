## The blocker (same as 2026-05-10, 2026-05-11, and 2026-05-12)

```
! [remote rejected] main -> main
  (refusing to allow a Personal Access Token to create or update
   workflow `.github/workflows/publish.yml` without `workflow` scope)
```

GitHub refuses, atomically, any push that creates or updates a
file under `.github/workflows/` if the PAT does not carry the
`workflow` scope (classic PAT) or *Actions: Read and write*
(fine-grained PAT). The PAT in `.gh-token` does not have it (the
original `SETUP.md` only asks for *Contents: Read and write* and
*Pull requests: Read and write*). Nothing else in the run failed.

## Decision

- The R14 commit was rolled back with `git reset --soft HEAD~1`
  followed by `git restore --staged .`, and the speculative
  `ROADMAP.md` / `CHANGELOG.md` edits were reverted with
  `git checkout`, so `main` does not falsely claim R14 shipped.
- The actual deliverable files
  (`.github/workflows/publish.yml`, `docs/release.md`,
  `tests/test_publish_workflow.py`, `mkdocs.yml`, `README.md`,
  `docs/index.md`, `docs/design.md`) are mirrored back to the
  Github folder so they are visible locally and ready for a
  human push after the PAT is updated.
- This work-log is the *only* file in today's commit, because a
  docs-only push does not touch `.github/workflows/` and therefore
  does not trip the scope check — so today's blocker is at least
  visibly recorded on `origin/main`, even when the R14 work itself
  cannot land.
