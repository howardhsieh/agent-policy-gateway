### Commit 1 — R14b (publish workflow)

- `.github/workflows/publish.yml` (106 lines): `name: Publish to
  PyPI`; triggers `push` on tags matching `v*` plus
  `workflow_dispatch`; top-level `permissions: contents: read`;
  three sequential jobs `test → build → publish` (each `needs:`
  the previous so the gate is strict).
    - `test` (ubuntu-latest): `actions/checkout@v4`,
      `actions/setup-python@v5` (Python 3.11),
      `pip install -e ".[dev]"`, `python -m ruff check .`,
      `python -m pytest`.
    - `build` (ubuntu-latest, `needs: test`): installs
      `build` + `twine`, runs `python -m build`,
      `python -m twine check --strict dist/*`, uploads `dist/`
      as an `actions/upload-artifact@v4` artifact named `dist`
      with `if-no-files-found: error` so a silent empty upload
      is a hard failure.
    - `publish` (ubuntu-latest, `needs: build`): targets the
      `pypi` GitHub environment (with the project URL), grants
      `permissions: id-token: write`, downloads the dist
      artifact via `actions/download-artifact@v4`, uploads with
      `pypa/gh-action-pypi-publish@release/v1` — no
      `PYPI_API_TOKEN` (or any other long-lived secret) is
      referenced anywhere in the file.

## The one remaining manual step

Today's run intentionally **does not** push a `v0.1.0` tag. The
PyPI side of trusted publishing — the "pending publisher" form on
<https://pypi.org/manage/account/publishing/> and the `pypi`
GitHub environment — is a one-time operator action that the
scheduled task cannot perform. Pushing the tag before that form
is filled in would trigger the workflow and fail the publish job
at the OIDC handshake, leaving a red CI run on the release commit.

When Howard is ready to actually publish v0.1.0 to PyPI, the
procedure is documented in `docs/release.md` §1 and §2 and reduces
to:

```bash
# 1. One-time PyPI configuration:
#    Open https://pypi.org/manage/account/publishing/
#    Add pending publisher:
#      PyPI project name: agent-policy-gateway
#      Owner:             howardhsieh
#      Repository name:   agent-policy-gateway
#      Workflow name:     publish.yml
#      Environment name:  pypi
#    On GitHub: Settings -> Environments -> New environment -> pypi
#
# 2. Tag and push:
git tag -a v0.1.0 -m "v0.1.0"
git push origin v0.1.0
#
# 3. Watch the workflow run under the Actions tab.
#    `test → build → publish` should all go green; the new
#    version appears on PyPI within a minute or two.
```

If the PyPI side cannot be configured for any reason, the manual
fallback documented in `docs/release.md` §3 (`python -m build` +
`python -m twine upload`) is a complete release path on its own —
the only difference is that an API token is used instead of OIDC.
