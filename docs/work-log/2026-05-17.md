## The blocker (recurrence of the R14 PAT-scope rejection)

```
! [remote rejected] main -> main
  (refusing to allow a Personal Access Token to create or update
   workflow `.github/workflows/publish.yml` without `workflow` scope)
```

GitHub refuses, atomically, any push that creates or updates a file
under `.github/workflows/` if the PAT does not carry the `workflow`
scope (classic PAT) or *Workflows: Read and write* (fine-grained
PAT). The PAT in `.gh-token` still does not have it — the same root
cause as 2026-05-10 / 2026-05-11 / 2026-05-12 / 2026-05-13 /
2026-05-16. Nothing else in the run failed.

The R14a design note in `docs/design.md` explicitly anticipated this:

> Still under *Up next* in the roadmap. The split exists because
> pushing a file under `.github/workflows/` requires a GitHub PAT
> carrying the `workflow` scope (classic) or *Workflows: Read and
> write* (fine-grained); the daily-task PAT currently does not
> carry that scope, so R14b cannot land via the scheduled run.

Today's run is the empirical confirmation, for the sixth time.

## Decision

- The local R14b commit (`feat(release): R14b — PyPI publish
  workflow + structural tests`) was rolled back with
  `git reset --hard origin/main` after the push rejection, so
  `main` does not falsely claim R14b shipped and the
  ROADMAP / CHANGELOG / README / `docs/design.md` /
  `docs/work-log/2026-05-17.md` edits do not get partially
  applied.
- This work-log is the **only** file in today's commit. A
  docs-only push does not touch `.github/workflows/` and
  therefore does not trip the scope check — so today's blocker is
  at least visibly recorded on `origin/main`, even when R14b
  itself cannot land.
- The prepared R14b files are recreatable from this work-log; the
  full diff is small (workflow file 106 lines, test file 323
  lines, plus small README / `docs/design.md` / ROADMAP /
  CHANGELOG edits) so re-staging once the PAT is fixed is not
  expensive.
